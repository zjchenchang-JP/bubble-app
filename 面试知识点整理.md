# 后端/运维面试知识点整理

## 一、Linux 常用命令

### 1.1 查看端口号

```bash
# 查看 8080 端口占用情况
netstat -tunlp | grep 8080

# 或者用 lsof
lsof -i :8080

# 或者用 ss（更快，推荐）
ss -tunlp | grep 8080
```

**参数说明：**
- `-t`: TCP 协议
- `-u`: UDP 协议
- `-n`: 以数字形式显示端口
- `-l`: 显示监听状态的 socket
- `-p`: 显示进程信息

---

### 1.2 查看进程状态

```bash
# 查看所有进程
ps -ef

# 另一种格式（更详细）
ps aux

# 实时监控进程
top

# 过滤特定进程
ps -ef | grep nginx

# 查看进程详情
cat /proc/<PID>/status

# 杀死进程
kill -9 <PID>        # 强制杀死
kill -15 <PID>       # 优雅退出

# 根据名称杀死进程
pkill -f process_name
killall process_name
```

---

### 1.3 查看磁盘空间

```bash
# 查看各分区使用情况
df -h

# 查看目录占用空间
du -sh /path

# 查看当前目录下各文件夹大小（排序）
du -sh * | sort -hr

# 查看 inode 使用情况
df -i
```

**参数说明：**
- `-h`: human-readable，以易读格式显示
- `-s`: summary，只显示总计
- `sort -hr`: 人类可读格式，逆序排序

---

### 1.4 其他常用命令

```bash
# 文件操作
ls -la           # 列出文件详情（包括隐藏文件）
cd / pwd / mkdir / rm -rf
cp / mv / touch

# 日志查看
tail -f log           # 实时查看日志
tail -n 100 log       # 查看最后 100 行
head -n 50 log        # 查看前 50 行
cat log | grep "error"

# 搜索
grep "keyword" file
grep -r "keyword" ./  # 递归搜索
find /path -name "*.log"

# 权限管理
chmod +x file         # 赋予执行权限
chmod 755 file        # rwxr-xr-x
chown user:group file

# 压缩解压
tar -czf archive.tar.gz dir/   # 打包
tar -xzf archive.tar.gz        # 解压
unzip file.zip

# 系统信息
uname -a         # 查看系统信息
free -h          # 查看内存使用
uptime           # 查看系统运行时间

# 网络相关
curl -I https://example.com    # 查看 HTTP 头
ping google.com
wget url
```

## 二、DNS 解析流程

### 2.1 DNS 是什么

DNS（Domain Name System）域名系统，将域名解析为 IP 地址。

### 2.2 完整解析流程

```
用户访问 www.example.com
       ↓
1. 检查浏览器缓存
       ↓ 没找到
2. 检查操作系统 hosts 文件 & 本地 DNS 缓存
       ↓ 没找到
3. 向本地 DNS 服务器（LDNS）发起查询
       ↓
4. LDNS 检查自身缓存
       ↓ 没找到
5. LDNS 向根域名服务器（.）查询
   → 返回 .com 的 NS 记录
       ↓
6. LDNS 向 .com 的 TLD 服务器查询
   → 返回 example.com 的 NS 记录
       ↓
7. LDNS 向 example.com 的权威 DNS 查询
   → 返回最终 IP 地址
       ↓
8. LDNS 返回给用户，同时缓存结果
```

### 2.3 DNS 记录类型

| 类型 | 作用 | 示例 |
|------|------|------|
| A | 域名指向 IPv4 地址 | `www.example.com → 1.2.3.4` |
| AAAA | 域名指向 IPv6 地址 | `www.example.com → 2001:db8::1` |
| CNAME | 域名别名 | `blog.example.com → example.com` |
| MX | 邮件服务器 | `@ → mail.example.com` |
| TXT | 文本记录（SPF、DKIM） | `@ → "v=spf1 include:..."` |
| NS | 域名服务器 | `@ → ns1.example.com` |

## 三、HTTP vs HTTPS

### 3.1 对比表格

| 对比项 | HTTP | HTTPS |
|--------|------|-------|
| 协议 | 应用层 | HTTP + SSL/TLS（传输层安全） |
| 端口 | 80 | 443 |
| 数据传输 | 明文，可被窃听/篡改 | 加密传输，安全 |
| 证书 | 不需要 | 需要 SSL 证书 |
| 性能 | 略快 | 略慢（TLS 握手开销） |
| SEO | 优势较低 | 搜索引擎优先收录 |
| 信任度 | 低 | 高（浏览器显示锁头图标） |
| 成本 | 免费 | 需要购买证书（Let's Encrypt 免费） |

### 3.2 HTTPS 握手流程

```
Client                                    Server
  │                                          │
  │─────── 1. ClientHello（支持的加密套件） ───→│
  │                                          │
  │←──── 2. ServerHello + 证书 + 选择加密套件 ──│
  │                                          │
  │←────── 3. ServerKeyExchange（可选） ───────│
  │                                          │
  │←──────────── 4. ServerHelloDone ──────────│
  │                                          │
  │  5. 验证证书                              │
  │  生成随机密钥                             │
  │                                          │
  │─────── 6. ClientKeyExchange（预主密钥） ──→│
  │                                          │
  │─────── 7. ChangeCipherSpec ──────────────→│
  │─────── 8. Finished（加密） ──────────────→│
  │                                          │
  │←────── 9. ChangeCipherSpec ───────────────│
  │←──────── 10. Finished（加密） ────────────│
  │                                          │
  │         11. 后续使用对称加密通信          │
```

### 3.3 为什么 HTTPS 握手后使用对称加密

| 加密方式 | 优点 | 缺点 |
|----------|------|------|
| 非对称（RSA/ECC） | 安全，无需共享密钥 | 速度慢，计算开销大 |
| 对称（AES/ChaCha20） | 速度快，效率高 | 需要安全地共享密钥 |

**混合方案：** 用非对称加密协商密钥，用对称加密传输数据

## 四、进程、线程和协程

### 4.1 对比表格

| 特性 | 进程 | 线程 | 协程 |
|------|------|------|------|
| 定义 | 资源分配的最小单位 | CPU 调度的最小单位 | 用户态轻量级线程 |
| 内存空间 | 独立内存空间 | 共享进程内存 | 共享，由用户控制 |
| 切换开销 | 大（涉及内核态切换） | 中等（内核态） | 极小（用户态，无内核干预） |
| 通信方式 | IPC（管道、消息队列、共享内存、Socket） | 共享内存/锁 | 内存共享，直接访问 |
| 创建/销毁 | 慢 | 较快 | 极快 |
| 数据隔离 | 是（进程间隔离） | 否（共享） | 否（共享） |
| 稳定性 | 高（一个崩溃不影响其他） | 中（一个崩溃可能影响整个进程） | 低（用户控制） |
| 编程复杂度 | 高（需要处理通信） | 中（需要处理同步） | 低（同步代码风格） |

### 4.2 形象比喻

```
进程 = 一家公司
  - 独立的办公室
  - 自己的资源
  - 公司之间需要正式沟通

线程 = 公司里的员工
  - 共享公司的资源
  - 员工之间可以共享文件
  - 需要协调避免冲突

协程 = 员工的多任务处理
  - 一个人在多个任务间快速切换
  - 看起来像同时做多件事
  - 切换成本极低
```

### 4.3 使用场景

| 场景 | 推荐 | 原因 |
|------|------|------|
| CPU 密集型 | 多进程 | 绕过 GIL，充分利用多核 |
| I/O 密集型 | 多线程/协程 | 等待时可以切换 |
| 高并发服务 | 协程 | 内存占用小，切换快 |
| 隔离任务 | 多进程 | 崩溃不影响其他 |

### 4.4 Go 语言中的 Goroutine

```go
// 协程示例
go func() {
    // 并发执行
}()

// Channel 通信
ch := make(chan int)
go func() {
    ch <- 42
}()
value := <-ch
```

## 补充：常见追问

### Q1: TCP 三次握手和四次挥手

```
三次握手（建立连接）：
Client                    Server
  │                          │
  │────── SYN ─────────────→│
  │                          │
  │←──── SYN+ACK ───────────│
  │                          │
  │────── ACK ─────────────→│
  │                          │
  ✓ 连接建立

四次挥手（断开连接）：
Client                    Server
  │                          │
  │────── FIN ─────────────→│
  │←───── ACK ──────────────│
  │                          │
  │←───── FIN ──────────────│
  │────── ACK ─────────────→│
  │                          │
  ✓ 连接关闭
```

### Q2: 输入 URL 到页面显示的全过程

1. DNS 解析域名 → IP
2. TCP 三次握手建立连接
3. TLS 握手（HTTPS）
4. 发送 HTTP 请求
5. 服务器处理并返回响应
6. 浏览器解析 HTML
7. 构建 DOM 树和 CSSOM 树
8. 合并生成渲染树
9. 布局和绘制
